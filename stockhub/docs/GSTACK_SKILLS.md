## 🎯 GStack 核心理念

**为什么需要 GStack？**
- ❌ 传统AI编码工具混合了所有模式，缺乏专业度
- ✅ GStack 提供明确的"齿轮切换"，按需求调用不同专业角色

**核心理念**:
- 规划 != 评审 != 发布
- 创始人品味 != 工程严谨度
- 每个角色需要不同的智能类型

## 🚀 6大工作流技能

### 1. /plan-ceo-review - 创始人/CEO模式

**什么时候用**: 每次开始新功能开发前

**目标**: 从用户角度重新思考问题，找到10星产品

**关键问题**:
- 这个请求背后真正的产品需求是什么？
- 这是显而易见的方案，还是隐藏了更好的方案？
- 如何让体验变得"必然、愉悦、甚至魔法"？

**示例**: 卖家上传照片功能
- ❌ 弱助手: 添加文件选择器，保存图片
- ✅ CEO模式:
  - 从照片识别产品
  - 搜索网络提取规格和定价
  - 自动起草标题和描述
  - 建议最佳主图
  - 检测照片质量
  - 创建高端体验

**StockHub应用场景**:
- 用户说"添加搜索功能" → CEO模式思考: 真正需求是帮助用户快速找到精准库存
- 用户说"添加物流追踪" → CEO模式思考: 真正需求是提供透明的物流体验和预计送达时间

### 2. /plan-eng-review - 工程经理模式

**什么时候用**: 产品方向确定后，开始开发前

**目标**: 锁定架构、数据流、边界情况、测试

**关键要素**:
- ✅ 架构设计
- ✅ 系统边界
- ✅ 数据流
- ✅ 状态转换
- ✅ 失败模式
- ✅ 边界情况
- ✅ 测试覆盖率
- ✅ **图表**（强制要求）

**要求的图表**:
- 架构图
- 状态机图
- 组件图
- 数据流图
- 测试矩阵

**StockHub应用场景**:
- 求购匹配流程: 上传 → 分类 → 丰富 → 起草 → 建议
- 哪些步骤同步？哪些异步？
- API边界在哪里？
- 失败重试逻辑？
- 防止重复任务？

### 3. /review - 悲观工程师模式

**什么时候用**: 代码完成后，合并PR前

**目标**: 找到CI通过但生产环境会失败的bug

**检查重点**:
- ❌ 不是风格检查
- ✅ N+1查询
- ✅ 读取陈旧数据
- ✅ 竞态条件
- ✅ 信任边界
- ✅ 缺失索引
- ✅ 转义bug
- ✅ 不变量破坏
- ✅ 错误的重试逻辑
- ✅ 测试通过但遗漏实际失败模式

**StockHub检查点**:
- 两个标签页是否可能覆盖主图选择？
- 失败的上传是否遗留孤立文件？
- 丰富化API部分失败时是否优雅降级？
- 从网络数据生成草稿是否存在prompt注入风险？

### 4. /ship - 发布工程师模式

**什么时候用**: 功能完成，准备发布

**目标**: 自动化发布流程，避免手动拖延

**任务清单**:
- ✅ 同步main分支
- ✅ 运行测试
- ✅ 检查分支状态
- ✅ 更新changelog或version
- ✅ 推送分支
- ✅ 创建或更新PR

**StockHub发布检查**:
```bash
# StockHub 发布流程
1. git checkout main && git pull
2. git checkout feature/xxx && git merge main
3. npm run build
4. npm test
5. npm version patch
6. git commit -am "chore: v1.x.y"
7. git push origin feature/xxx
8. gh pr create --title "Feature: xxx"
```

### 6. /retro - 工程经理模式

**什么时候用**: 周末或定期回顾

**目标**: 分析提交历史、工作模式、发布速度

**分析内容**:
- 提交频率趋势
- 代码审查模式
- 发布速度
- 工作效率
- 阻塞因素

**StockHub回找示例**:
- 本周完成了多少功能？
- 平均每个功能从规划到发布需要多久？
- 哪些步骤最耗时？
- 如何优化？

## 📋 StockHub 应用工作流

### 功能开发标准流程

#### 步骤1: /plan-ceo-review (5-10分钟)
```
用户: "添加搜索功能"
→ CEO模式: 真正需求是什么？
→ 答案: 快速找到精准库存 + 智能推荐
→ 规划: 多维度搜索 + AI推荐 + 搜索历史
```

#### 步骤2: /plan-eng-review (10-20分钟)
```
→ 工程模式: 技术架构
→ 图表: 搜索架构图、数据流图
→ 测试矩阵: 搜索条件 × 数据类型 × 结果排序
→ 边界: 空结果、特殊字符、超长查询
```

#### 步骤3: 开发 (按计划实现)

#### 步骤4: /review (5-10分钟)
```
→ 检查: N+1查询、竞态、边界情况
→ 验证: 索引、缓存、错误处理
→ 测试: 单元测试、集成测试
```

#### 步骤5: /ship (2-3分钟)
```
→ 自动: 同步、测试、推送、PR
→ 手动: 发布说明、版本标记
```

#### 步骤6: /browse (1-2分钟)
```
→ QA测试: 完整功能验证
→ 截图: 每步验证
→ 检查: console错误、布局问题
```

#### 步骤7: /retro (周末)
```
→ 分析: 工作效率、发布速度
→ 优化: 改进流程、消除瓶颈
```

## 💡 学习心得

- **明确切换**: 明确告诉AI当前使用哪个模式
- **完整闭环**: 从规划到发布到QA的完整流程
